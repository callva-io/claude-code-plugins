# Idempotency — Dedup by Stable External ID

Any automation that pulls records from an external system and creates them in CallVA must be **idempotent**: running it twice should produce the same database state as running it once, not duplicates. Without this, re-runs after a failure, catch-up batches, and scheduled overlaps silently corrupt data.

## The requirement: a stable external ID

Every incoming record must have an ID that:

- Is assigned by the external system (not generated by your script)
- Is unique within the external system
- Survives across runs — the same record produces the same ID every time you see it

Common sources:

- `appointment.uuid` from a clinic booking system
- `order_id` from an e-commerce platform
- `ticket_id` from a CRM
- `event_id` from a webhook provider

If the external system exposes no stable ID, you cannot do proper dedup. You can construct a synthetic key from multiple fields (`phone + date + doctor`), but any field you include in that key that can change over time will break the dedup. Treat synthetic keys as a last resort.

## Store the external ID on the CallVA record

When you create a CallVA call (or whatever resource), include the external ID as a first-class field:

```typescript
await callvaRequest("POST", "calls", {
  appointment_id: appt.uuid, // external ID preserved verbatim
  phone: appt.phone,
  // ... other fields ...
});
```

CallVA's `appointment_id` field accepts any string. For other resource types, use whichever field the schema provides for external correlation. Never encode it into a compound field like `notes` or `description` — it must be queryable.

## The dedup pattern: `Set<externalId>`

Fetch the existing CallVA records in your target window, build a `Set` from their external IDs, and filter incoming records against it. Also add to the set as you go, so duplicates *within* the current batch are also caught:

```typescript
// 1. Load existing records in the target window from CallVA
const existingResult = await callvaRequest(
  "GET",
  `calls?call_at_gte=${today}T00:00:00Z&per_page=500`,
);
const existingCalls = existingResult?.data ?? [];

// 2. Build a Set of known external IDs (drop null/undefined)
const existingApptIds = new Set(
  existingCalls.map((c) => c.appointment_id).filter(Boolean),
);

// 3. Filter incoming records; mutate the set as you go to
//    prevent duplicates WITHIN the current batch
const toCreate = incoming.filter((item) => {
  if (existingApptIds.has(item.appointment_id)) return false;
  existingApptIds.add(item.appointment_id);
  return true;
});
```

The two failure modes the `Set.add` inside the filter prevents:

- **External duplicates** — same ID already in CallVA from a prior run (caught by the initial set population).
- **Batch-internal duplicates** — same ID appearing twice in the current fetch from the external source (caught by `existingApptIds.add` inside the filter).

## Proving it works: re-run observation

Run the automation once, creating N records. Run it again immediately with no source-side changes. The result should show:

- `raw`: N (fetched the same N from source)
- `duplicates`: N (all blocked by dedup)
- `created`: 0 (nothing written)

This is the property you want. Any automation where the second consecutive run creates more records is broken and needs the dedup fixed before it's allowed anywhere near production.

## Report `duplicates` as a first-class count

In the return value, surface `duplicates` as one of the outcome categories, not as a silent "we skipped some":

```typescript
return {
  appointments: {
    raw: 339,
    created: 1,
    duplicates: 327,
    blocked: 2,
    filtered: 10,
    create_failed: 0,
  },
  // ...
};
```

This lets the operator distinguish "the automation ran cleanly and found nothing new" from "the automation created 50 new records" at a glance.

## Pagination gotcha

If your existing records might exceed the CallVA API's `per_page` limit, you need to paginate the fetch or scope the time window tight enough to fit. **Partial existing-set data silently allows duplicates** — dedup only protects against what's in the set at filter time.

Best practice: scope the existing-fetch query to the same window as the incoming data. For a daily scheduler that processes tomorrow's appointments, the existing-fetch only needs records for tomorrow's call window — never more than a few hundred records, always fits in one page.

## When this doesn't apply

- **Webhook handlers** that react to a single event don't need batch-level dedup — the webhook provider is responsible for at-least-once or exactly-once delivery semantics. If it's at-least-once, you still need per-event dedup by event ID, but the pattern is a single-record check, not a batch filter.
- **Pure transforms** that read from CallVA, mutate, and write back — there's no external source, so there's nothing to dedup against.
- **One-shot maintenance scripts** intended to run exactly once, with human oversight, against a fixed dataset.
